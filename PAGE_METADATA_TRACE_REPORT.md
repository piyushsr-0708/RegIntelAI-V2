## Root Cause Analysis

### Why Page Numbers Are Corrupted:

1. **Backend provides correct data:**
   - `requirements[].page_numbers = [1, 2]` (from parsed PDF)

2. **Frontend IGNORES backend data for sessions:**
   - Uses simulation pipeline that generates random `source_page` values
   - Only uses backend data to display counts, not individual requirement metadata

3. **Frontend should use:**
   ```javascript
   source_page: req.page_numbers ? req.page_numbers[0] : null
   ```

4. **But instead uses:**
   ```javascript
   source_page: 1 + Math.floor(rng() * Math.max(1, safeSegments.length * 3))
   ```

## LLM Prompt Analysis

The page metadata is **NOT generated by LLM**. It is:

1. **Parsed from PDF** by `pipeline/parser/pdf_parser.py`
2. **Propagated through stages** as `page_number` → `page_range` → `page_numbers`
3. **Lost at MAP stage** (not included in MAP schema)
4. **Never ingested to database** (models lack page columns)
5. **Randomly generated by frontend** for demo purposes

No LLM prompt is involved in page number generation - the parser extracts real page numbers from PyMuPDF, but they're lost before reaching the UI.

---

## Conclusion

**The "Source Page" displayed in Session Dashboard is NOT FROM THE PIPELINE.**

It's a randomly generated value from `frontend/src/pipeline/stages/stageExtraction.js:38` used for frontend demo simulation.

The actual page metadata (`page_numbers` arrays) exists correctly in:
- ✓ requirements JSON files
- ✓ Backend API responses
- ❌ NOT in database models
- ❌ NOT used by frontend
- ❌ NOT displayed to users

**To fix:** Frontend should use `req.page_numbers[0]` from backend data instead of generating random values.
